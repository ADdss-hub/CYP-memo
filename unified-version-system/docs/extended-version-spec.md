# 扩展版本规范

## 版本格式

采用语义化版本控制规范 (SemVer)，版本号格式为 `vMAJOR.MINOR.PATCH`：

- **MAJOR** 主版本号：当你做了不兼容的API修改时递增
- **MINOR** 次版本号：当你做了向下兼容的功能性新增时递增
- **PATCH** 修订号：当你做了向下兼容的问题修正时递增

额外标签标识：
- `-alpha.N` Alpha版本
- `-beta.N` Beta版本
- `-rc.N` 候选版本

示例：
- v1.0.0
- v2.1.3-beta.1
- v3.0.0-alpha.2

## 变更类型规范

为了更好地跟踪和管理版本变更，我们采用约定式提交（Conventional Commits）规范来定义变更类型。每条变更记录必须包含一个类型，用于指示变更的性质。

### 标准变更类型

| 类型       | 说明                         | 对版本号的影响        |
|------------|------------------------------|----------------------|
| feat       | 新功能                       | MINOR 版本递增        |
| fix        | 修复问题                     | PATCH 版本递增        |
| docs       | 文档更新                     | PATCH 版本递增        |
| style      | 代码风格更新（不影响功能）   | PATCH 版本递增        |
| refactor   | 代码重构（不影响功能）       | PATCH 版本递增        |
| perf       | 性能优化                     | PATCH 版本递增        |
| test       | 测试相关                     | PATCH 版本递增        |
| build      | 构建系统或外部依赖变更       | PATCH 版本递增        |
| ci         | CI配置文件和脚本变更         | PATCH 版本递增        |
| chore      | 日常维护工作                 | PATCH 版本递增        |
| revert     | 回滚之前的提交               | 根据被回滚的提交决定  |

### 破坏性变更类型

| 类型       | 说明                         | 对版本号的影响        |
|------------|------------------------------|----------------------|
| BREAKING CHANGE | 不兼容的API修改         | MAJOR 版本递增        |

破坏性变更可以在任何类型的提交中声明，只要提交信息中包含 `BREAKING CHANGE:` 字段。

## 版本变更规则

### PATCH 版本号递增
适用于以下变更类型：
- fix（修复问题）
- docs（文档更新）
- style（代码风格更新）
- refactor（代码重构）
- perf（性能优化）
- test（测试相关）
- build（构建系统或外部依赖变更）
- ci（CI配置文件和脚本变更）
- chore（日常维护工作）

### MINOR 版本号递增
适用于以下变更类型：
- feat（新功能）
- 任何包含 `BREAKING CHANGE:` 但不改变API的变更

### MAJOR 版本号递增
适用于以下变更类型：
- 任何包含 `BREAKING CHANGE:` 且改变了API的变更

## 版本发布原则

1. 稳定版本不能跟随预发布版本
2. 预发布版本可以跟随稳定版本
3. 版本号必须是数字，禁止使用字母等其他字符
4. 版本号递增必须是单调递增的
5. 每次版本变更必须记录相应的变更类型和描述
6. 破坏性变更必须明确标记并在变更日志中详细说明
